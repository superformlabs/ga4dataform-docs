Full Custom Config file (for easy copying)

Description: Below is a full, ready-to-copy, Premium compatible custom config file (includes/custom/modules/ga4/config.js) that you can use if you updated from Core (since we don't overwrite config.js files during updates).

Existing variables

Remember to save your current variables before proceeding - you'll need to restore them afterward.

// do not remove this line
// eslint-disable-next-line @typescript-eslint/naming-convention
const { helpers } = require("../../../core/helpers");

// eslint-disable-next-line @typescript-eslint/naming-convention, no-unused-vars
const lowerSQL = helpers.lowerSQL;

/*
    ga4dataform runs the core model with SQL that can be tweaked with
    configuration settings in this file.

    Below, you will find a sample config file that you can tweak to
    your likings.

    See the documentation for all details.

    There are more configuration settings than in this sample file.
    See core/config.js for all config parameters
*/

GA4 Start Date
GA4_START_DATE: "2020-01-01"

Description: Defines the starting date for the GA4 data that the model should process.
Customization: Change this date to reflect the earliest data you want to include in your reports. For example, if you started collecting GA4 data on a different date, modify this to match that date.

Use Fresh Events 
USE_FRESH_EVENTS: true

Description: If you set this to true, ga4_events will query from fresh table shards if they are available. If events_* table also exists for the same date, that will be used. Priority for same date shards: 1. regular events_ 2. fresh_ (if enabled) 3. intraday_ (if enabled)

Use Intraday Events 
USE_INTRADAY_EVENTS: false

Description: If you set this to true, ga4_events will query from intraday table shards if they are available. Priority for same date shards: 1. regular events_ 2. fresh_ (if enabled) 3. intraday_ (if enabled)

Multi-Property Support 
  EXTRA_GA4_DATASETS_ARRAY: []

Description: Enables combining multiple GA4 properties into a single unified data model by UNIONing their events tables. All datasets must be in the same GCP project and BigQuery region. The main property (defined in workflow_settings.yaml) is automatically included, so only specify additional properties here. Each property is identified by a property_id column in the output.

Requirements:

  • All datasets must be in the same BigQuery region
  • All datasets must be in the same GCP project
  • Same configuration applies to all properties

Configuration: Add dataset names as strings in an array format.

Example:

  EXTRA_GA4_DATASETS_ARRAY: [
    "analytics_234567890",
    "analytics_345678901",
    "analytics_456789012"
  ]

Output: The resulting ga4_events table will include a property_id column that contains the dataset name, allowing you to distinguish which property each event originated from. All properties share the same event filtering, hostname rules, and custom parameter configurations.

Custom Event Parameters
CUSTOM_EVENT_PARAMS_ARRAY: []

Description: This allows you to specify custom event parameters that are not part of the standard GA4 data. The custom parameters will be added to the event_params_custom column in the ga4_events table.
Customization: Add parameters using the format { name: "paramname", type: "TYPE", renameTo: "outputcolumnname", description: "This parameter means XYZ" }.

FieldDescription
nameRequired. The original event parameter name to extract
typeRequired. Data type for extraction and storage. int: Extracts from int_value, stores as INT; string: Extracts from all values, stores as STRING; decimal: Extracts from all numerical values, stores as FLOAT
renameToOptional. New name for the output column. If not specified, original parameter name is used
cleaningMethodOptional. Transformation method to apply: lowerSQL: Converts string outputs to lowercase
descriptionOptional. Add a description for a column that will show up in the table schema

Example:

CUSTOM_EVENT_PARAMS_ARRAY: [
  { name: "gtm_info", type: "string", description: "GTM container ID + container version"},
  { name: "consent", type: "string", description: "The current consent status of the user"},
  { name: "page_type", type: "string", description: "The type of page: category, listing, product, home, checkout" }
]

FieldDescription

renameToOptional. New name for the output column. If not specified, original parameter name is used
cleaningMethodOptional. Transformation method to apply: lowerSQL: Converts string outputs to lowercase
descriptionOptional. Add a description for a column that will show up in the table schema

Example:

  CUSTOM_USER_PROPERTIES_ARRAY: [
  { name: "user_tier", type: "string", description: "The loyalty tier of the user (e.g., 'gold', 'silver')" },
  { name: "lifetime_value", type: "decimal", renameTo: "ltv", description: "The calculated lifetime value of the user" },
  { name: "first_purchase_date", type: "string", description: "The date of the user's first purchase" }

Custom Session Parameters 
  CUSTOM_SESSION_PARAMS_ARRAY: []

Description: Include session-level custom parameters in the sessions table. These will appear in int_ga4_sessions and ga4_sessions tables in the session_params_custom column.

Customization: Add parameters with aggregation methods, event filters, and parameter filters to select specific values per session.

Configuration Fields

FieldDescription
nameRequired. The full path to the parameter to aggregate (e.g., event_params_custom.lead_id). It should exist in the ga4_events table. Can include SQL functions like safe_cast().
pickRequired. The aggregation method: first (chronologically first value), last (chronologically last value), min (minimum value), max (maximum value), boolean (true if parameter exists with non-null value).
renameToOptional. New name for the output column. If not specified, the parameter name is used and automatically sanitized to a valid BigQuery column name.
eventFilterOptional. Filter by event name(s). Can be a single string (e.g., "page_view") or an array (e.g., ["page_view", "screen_view"]). Only events matching the filter will be considered.
paramFilterOptional. Filter by parameter conditions. See Parameter Filtering section below for detailed syntax.
ignoreNullsOptional. Controls null handling. Default is true for first/last (skips nulls), always true for max/min/boolean. Set to false to include null values in first/last aggregations.
cleaningMethodOptional. Transformation method to apply: lowerSQL converts string outputs to lowercase.
descriptionOptional. Column description that will appear in the table schema.

Parameter Filtering

The paramFilter field allows you to filter events based on parameter values before aggregation.

Single Condition:

paramFilter: {
  param: "event_params_custom.page_type",  // parameter to check
  value: "checkout",                        // value to match
  operator: "="                             // optional, defaults to "="
}

Supported Operators:

  • =, != - equality/inequality
  • >, >=, <, <= - comparisons (works with numbers)
  • in, not in - list matching (value should be an array)
  • like, not like - pattern matching
  • is null, is not null - null checks (no value needed)

Multiple Conditions with AND/OR Logic:

paramFilter: {
  conditions: [
    { param: "event_params_custom.item_category", value: ["electronics", "computers"] },
    { param: "event_params_custom.user_country", value: "US" },
    { param: "safe_cast(event_params_custom.price as numeric)", value: 500, operator: ">=" }
  ],
  logic: "and"  // or "or" - defaults to "and"
}

Examples

Basic aggregation (no filters):

CUSTOM_SESSION_PARAMS_ARRAY: [
  { name: "event_params_custom.user_id", pick: "first", description: "First user ID" },
  { name: "event_params_custom.page_location", pick: "last", renameTo: "exit_page", description: "Last page viewed" },
  { name: "safe_cast(event_params_custom.scroll_depth as int64)", pick: "max", renameTo: "max_scroll", description: "Maximum scroll depth" },
  { name: "event_params_custom.conversion_flag", pick: "boolean", renameTo: "had_conversion", description: "Whether any conversion occurred" }
]

Event filtering:

CUSTOM_SESSION_PARAMS_ARRAY: [
  // single event filter
  { 
    name: "event_params_custom.page_title", 
    pick: "first", 
    renameTo: "landing_page_title",
    eventFilter: "page_view",
    description: "First page title" 
  },
  
  // multiple event filter
  { 
    name: "event_params_custom.utm_source", 
    pick: "last", 
    renameTo: "last_traffic_source",
    eventFilter: ["page_view", "screen_view"],
    description: "Last traffic source from navigation events" 
  }
]

Null handling:

CUSTOM_SESSION_PARAMS_ARRAY: [
  // ignore nulls (default behavior)
  { 
    name: "event_params_custom.lead_id", 
    pick: "first", 
    renameTo: "first_lead_id",
    eventFilter: "page_view",
    ignoreNulls: true,  // can be omitted, this is default
    description: "First non-null lead ID" 
  },
  
  // respect nulls - include null values
  { 
    name: "event_params_custom.lead_id", 
    pick: "first", 
    renameTo: "initial_lead_id_raw",
    eventFilter: "page_view",
    ignoreNulls: false,  // will return null if first value is null
    description: "First lead ID including nulls" 
  }
]

Single parameter filters:

CUSTOM_SESSION_PARAMS_ARRAY: [
  // equality filter
  { 
    name: "event_params_custom.campaign_id", 
    pick: "first", 
    renameTo: "checkout_campaign",
    eventFilter: "page_view",
    paramFilter: {
      param: "event_params_custom.page_type",
      value: "checkout"
    },
    description: "Campaign ID from first checkout page" 
  },
  
  // IN operator with array
  { 
    name: "event_params_custom.product_id", 
    pick: "last", 
    renameTo: "last_tech_product",
    eventFilter: "view_item",
    paramFilter: {
      param: "event_params_custom.item_category",
      value: ["electronics", "computers", "mobile"]
    },
    description: "Last tech product viewed" 
  },
  
  // comparison operator with numbers
  { 
    name: "safe_cast(event_params_custom.cart_value as numeric)", 
    pick: "max", 
    renameTo: "max_high_value_cart",
    eventFilter: "add_to_cart",
    paramFilter: {
      param: "safe_cast(event_params_custom.cart_value as numeric)",
      value: 100,  // number without quotes

Custom Session Totals 
CUSTOM_SESSION_TOTALS: {
    eventsToCount: [],
    uniqueFields: [],
    sumFields: [],
  }

Description: Provide a list of fields you want to aggregate at the session level. These will end up in the int_ga4_sessions ga4_sessions table in the corresponding struct within totals (e.g., totals.count, totals.distinct_count, totals.has_events, totals.sum).

Customization: Configure the eventsToCount, uniqueFields, and sumFields arrays. You can use a simple string for the name or an object for more complex logic like renaming, filtering, or grouping events.

note

Column descriptions are auto-generated for session total fields based on the configuration!

Each array in CUSTOM_SESSION_TOTALS supports multiple format options to accommodate different use cases. Choose the format that best fits your needs!

Events to Count (eventsToCount)

Counts occurrences of specific events per session. Results are stored in totals.count.* and totals.has_event.* in int_ga4_sessions and ga4_sessions tables.

Supported Formats:

FormatUse CaseExample
StringSimple event counting'page_view'
Object with metadataEvent counting with custom naming{ name: 'sign_up', renameTo: 'registrations' }
Object with event arrayCount multiple related events as one metric{ name: ['video_start', 'video_progress'], renameTo: 'video_interactions' }

Available Fields:

  • name (required): Event name(s) to count - string or array of strings
  • renameTo (optional): Custom output column name
Unique Fields (uniqueFields)

Counts distinct values of specified fields per session. Results are stored in totals.distinct_count.* in int_ga4_sessions and ga4_sessions tables.

Supported Formats:

FormatUse CaseExample
Object with field pathCount unique values across all events{ name: 'ecommerce.transaction_id', renameTo: 'unique_transactions' }
Object with event filterCount unique values only from specific events{ name: 'event_params.currency', eventFilter: ['purchase', 'refund'], renameTo: 'transaction_currencies' }

Available Fields:

  • name (required): Field path to count unique values from
  • eventFilter (optional): Limit counting to specific event types - string or array of strings
  • renameTo (optional): Custom output column name
Sum Fields (sumFields)

Sums numeric values of specified fields per session. Results are stored in totals.sum.* in int_ga4_sessions and ga4_sessions tables.

Supported Formats:

FormatUse CaseExample
StringSimple field summation across all events'event_params.video_duration'
Object with metadataField summation with custom naming{ name: 'event_params.engagement_time_msec', renameTo: 'total_engagement' }
Object with event filterSum values only from specific events{ name: 'ecommerce.purchase_revenue', eventFilter: ['purchase', 'refund'], renameTo: 'net_purchase_revenue' },

Available Fields:

  • name (required): Field path to count unique values from
  • eventFilter (optional): Limit counting to specific event types - string or array of strings
  • renameTo (optional): Custom output column name

Example:

CUSTOM_SESSION_TOTALS: {
  eventsToCount: [
    'page_view',
    { name: 'sign_up', renameTo: 'registrations', description: 'User registration events in session' },
    { name: ['video_start', 'video_progress', 'video_complete'], renameTo: 'video_engagement', description: 'All video interaction events' }
  ],
  uniqueFields: [
    { name: 'ecommerce.transaction_id', renameTo: 'unique_transactions', description: 'Number of unique transaction IDs in session' },
    { name: 'event_params.currency', renameTo: 'funnel_currencies', eventFilter: ['begin_checkout', 'purchase'], description: 'Unique currencies used in checkout funnel' },
  ],
  sumFields: [
    { name: 'event_params.engagement_time_msec', renameTo: 'total_engagement_time', description: 'Sum of engagement time in milliseconds' },
    { name: 'ecommerce.purchase_revenue', eventFilter: ['purchase', 'refund'], renameTo: 'net_purchase_revenue' },
  ]
}
Item fields are not (yet) supported!

This will throw an error:
{ name: "items.item_params_custom.item_sku", eventFilter: ['purchase'] }

Custom Channel Grouping 
  CUSTOM_CHANNEL_GROUPING: []

Description: Define custom channel groupings that will be added to:

  • int_ga4_sessions: session_source.first.{groupingName} and session_source.last.{groupingName}
  • ga4_sessions: session_source.{groupingName} and last_non_direct_traffic_source.{groupingName}

Custom channel groupings allow you to create your own traffic source classifications beyond GA4's default channel grouping. You can either build groupings from scratch or override specific channels while keeping the rest as default.

Configuration Structure:

FieldDescription
groupingNameRequired. Name of the output column that will appear in your sessions tables
descriptionOptional. Documentation text that appears in the BigQuery table schema
useDefaultChannelGroupingAsFallbackOptional. When true, only define channels you want to override. All other traffic falls back to default_channel_grouping. When false or omitted, unmatched traffic becomes '(Other)'.
customColumnsOptional. Array of custom fields to reference in your criteria. Each entry has name (reference name used in criteria) and path (full column path like 'device.category')
channelDefinitionsRequired. Array of channel rules evaluated in order (first match wins). Each has name (channel name) and criteria (SQL expression defining when this channel applies)

Available Default Fields: source, medium, campaign, category, term, content, campaign_id, default_channel_grouping

Default Channel Grouping Field

By default, we use the "manually" created session_source.first.default_channel_grouping and session_source.last.default_channel_grouping as the Default Channel Grouping input for the custom channel groupings, NOT Google's session_traffic_source_last_click.cross_channel_campaign.default_channel_group. If you wish to use that instead, you need to add it as a customColumn and use that in the criteria!

Custom Columns: Reference any field from your data model by defining them in the customColumns array. Common examples:

  • Device: device.category
  • Geography: geo.country
  • Landing_page: landing_page_location, landing_page_referrer
  • Custom session parameters you defined in CUSTOM_SESSION_PARAMS_ARRAY

Criteria Syntax:

  • Use standard BigQuery SQL syntax and functions
  • Order matters: First matching rule wins (like a CASE WHEN statement)
  • Common patterns:
    • Exact match: source = 'google'
    • Multiple values: medium in ('cpc', 'ppc', 'paid')
    • Pattern matching: regexp_contains(source, r"^(facebook|instagram)$")
    • Null checks: coalesce(source, medium) is null
    • Combinations: source = 'google' and medium = 'cpc'

Examples

Example 1: Copy default grouping

  CUSTOM_CHANNEL_GROUPING: [
  {
    groupingName: 'paid_search_split',
    description: 'Split paid search by search engine, keep all other channels as default',
    useDefaultChannelGroupingAsFallback: true,
    channelDefinitions: [
      { name: 'Google Paid Search', criteria: `source = 'google' and category = 'SOURCE_CATEGORY_SEARCH' and regexp_contains(medium, r"^(cpc|ppc|paid)$")` },
      { name: 'Bing Paid Search', criteria: `source = 'bing' and category = 'SOURCE_CATEGORY_SEARCH' and regexp_contains(medium, r"^(cpc|ppc|paid)$")` },
      { name: 'Other Paid Search', criteria: `category = 'SOURCE_CATEGORY_SEARCH' and regexp_contains(medium, r"^(cpc|ppc|paid)$")` }
    ]
  },
  {
    groupingName: 'device_geo_grouping',
    description: 'Channel grouping segmented by device type and geography',
    useDefaultChannelGroupingAsFallback: true,
    customColumns: [
      { name: 'device_category', path: 'device.category' },
      { name: 'geo_country', path: 'geo.country' }
    ],
    channelDefinitions: [
      { name: 'Mobile Direct', criteria: `device_category = 'mobile' and source = 'direct' and medium = '(none)'` },
      { name: 'Desktop Paid Search - US', criteria: `device_category = 'desktop' and geo_country = 'United States' and category = 'SOURCE_CATEGORY_SEARCH' and regexp_contains(medium, r"^(cpc|ppc|paid)$")` },
      { name: 'Paid Social', criteria: `(regexp_contains(source, r"^(facebook|instagram|twitter)$") or category = 'SOURCE_CATEGORY_SOCIAL') and regexp_contains(medium, r"^(cpc|ppc|paid)$")` },
      { name: 'Email', criteria: `regexp_contains(medium, r"newsletter|mail")` }
    ]
  }
],
Best Practices
  • Use useDefaultChannelGroupingAsFallback: true to avoid redefining all standard channels
  • Add description fields to document your grouping logic in the schema
  • Test your criteria expressions on a subset of data first
  • Remember that order matters - more specific rules should come before general ones
  • Use customColumns sparingly - only when standard traffic fields aren't sufficient

User stitching 
USER_STITCHING: {
  enabled: false,
  dimensions: [],
  lookBackDays: 90,
  lookAheadDays: 10
}

Description: The user stitching functionality looks at the cookie of the user (user_pseudo_id) and "fills in gaps" of certain column(s) - most notably the user_id. It does this on the session level.

How it works:

  • Lookback: we look back in time to find if that visitor in the past has had a session with a user_id, and assign that user_id to the current session. The last seen user_id is used to fill in the blanks.
  • Lookahead: if there is no session in the past with a user_id, we look in the future to find one. This effectively rewrites the history and de-anonimises sessions leading up to a login (caution is advised not to breach privacy regulations, policies).
  • Results are stored in the stitched_ids struct column in the ga4_sessions table.
  • The stitching is not restricted to user_id only: if you have (custom) parameters, user_properties, url_params, you can stitch them via the same procedure using the dimensions array.
  • We use user_pseudo_id as the partition for the lookback and lookahead windows

Configuration Options:

FieldDescription
enabledEnable/disable user stitching functionality
dimensionsArray of objects that contain additional fields to stitch beyond user_id
lookBackDaysDays to search backwards for user identification (default: 90)
lookAheadDaysDays to search forward for retroactive identification (default: 10)

Dimensions Format:

  • name: the column name as it appears in ga4_events
  • renameTo: the column name as it will appear in the stitched_ids struct in ga4_sessions
Field naming

If no renameTo field is used, the full (sanitized) name is used. So event_params_custom.lead_id will become event_params_custom_lead_id.

Example:

  USER_STITCHING: {
    enabled: true,
    dimensions: [
	    { name: "user_properties.datatrics_segment", renameTo: "datatrics_segment" },
      { name: "event_params_custom.showroom_name" }
    ],
    lookBackDays: 90,
    lookAheadDays: 3
  }
Respect user privacy

Be mindul of user choices and privacy when stitching data. If you use stitched data to enrich customer profiles with information gathered while they were logged out, it may not be what they expect. Or even what you promised in your own privacy policy.

Custom Event ID Parameter 
CUSTOM_EVENT_ID_PARAMETER: ''

Description: Use CUSTOM_EVENT_ID_PARAMETER to add another custom event parameter that makes an event unique. This field will be concatenated with the default event_id fields and will ensure event uniqueness. We usually recommend using a UNIX timestamp or a custom event_id.

Example:

CUSTOM_EVENT_ID_PARAMETER: 'event_params_custom.hit_timestamp'

Which will return this SQL in ga4_events:

farm_fingerprint(
    event_name || '-' ||
    time.event_timestamp || '-' ||
    coalesce(user_pseudo_id, 'no') || '-' ||
    coalesce(event_params.ga_session_id, 0) || '-' ||
    coalesce(batch.batch_page_id, 0) || '-' ||
    coalesce(batch.batch_ordering_id,0) || '-' ||
    coalesce(batch.batch_event_index,0) || '-' ||
    coalesce(ecommerce.ecommerce_transaction_id, "no") || '-' ||
    coalesce(event_params.engagement_time_msec, 0)
    || safe_cast(coalesce(event_params_custom.hit_timestamp,'no') as string)
  ) as event_id
warning

Make sure to add the custom field you are using as CUSTOM_EVENT_ID_PARAMETER to the CUSTOM_EVENT_PARAMS_ARRAY as well!
